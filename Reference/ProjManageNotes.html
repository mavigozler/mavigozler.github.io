Project Charter

This is the "vision" document and should contain following:

  • name of project
  • purpose of project and business case
  • project team, including sponsor, key business stakeholders
  • project approach, with technologies chosen and why
  • project goal date & high-level milestones
  • cost assumptions based on research

"Final discovery"

This is more detailed research to have information to put together the scope and plan for the project. This can take from 1-3 months for a $500K-$1M project; for a $5-20M range project, 6-12 months. For a $250-500K project, 6-8 weeks. For < $250K, a month or less.

This work will produce the "scope document" and "business requirements document (high-level)". It is called Scope of Work by vendors. The points addressed are:

  • What will specifically be done?
  • System/application features and functions
  • Features and functions excluded from work
  • Who is doing what: roles
  • Dependencies---this is part of risks
  • Order of doing things (part of dependencies)
  • Fitting a timeline of activity
  • Unknowns and risks
  • the budget

This is the "map", but not a detailed plan.

"The plan for the plan"
Consider a project to roll out a new website with new CMS, involving a data migration and a integration to back end with customer and financial data. Multiple teams are involved. These are the elements of the plan for this project:

  • Hosting % identify hosting partner % contract, licensing, negotiations % plan for integrations % test for performance % dependencies - multiple, need integration/performance testing % timeline: # July-Aug: plan identify # Sept: environment ready # Dec: load testing, performance testing
  • Content Entry % what need over and above CMS content entry interface? % workflow # approvals % taxonomy and tagging % dependencies - content editing and entry, UAT % timeline: # July/Aug: requirements # Sept/Oct: build # Nov: UAT
  • Search
  • Content Pages and Features
  • Integrations
  • Back-end System
  • Data Migration

Each of the remaining sections had some of the same detail. What this multi-level outline list is basically a brainstorm. it was refined later.

This was a one-person listing by an experienced person. Normally a team will be involved in the brainstorm. Questions to be asked are:

  • what are the components of the project: these are the highest level
  • Now try to itemize at the 2nd level
  • what dependencies are there between the components?
  • When would each component need to be completed to achieve the goal date?

This meeting should involve asking the question "what else?" several times. Write everything down. It can be discarded later.

Project Scope & Plan (definitions, documentation)

At this point of the project, detailed business requirements and the "plan for the plan" should be on paper. Scope is about the project deliverables

  • project summary taken from charater, a paragraph or two, business case for project
  • project deliverables identified in breakout meetings and plan for the plan notes
  • what is out of scope important to identify this so stakeholders dont make assumptions
  • constraints this is a "fixed object" that is not going to change. This would be like rules/laws/regulations of government affecting the project
  • assumptions examples are when customer accepts project delivery date, when vendor can deliver their part by a certain date, project decisions not overturned by other business decisions, some piece of tech works as advertised
  • risks examples are dependencies on 3rd parties work/tech, absolute date affecting a project (schedule risk)
  • timeline this is either a bottom-up or top-down dtermination. If business is flexible, the team determines the timeline in bottom-up fashion. But when business sets a firm date, the timeline is top-down.
  • budget
  • success metrics This part includes discussing return on investment. Metrics are called key performance indicators (KPIs).
